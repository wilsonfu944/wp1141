# LINE Bot 對話/功能設計文件

## 1. 主題

**戀愛機器人（Love Bot）** - 一個扮演使用者男女朋友的 AI 聊天機器人，提供陪伴、關心和支持。

## 2. 功能列表

### 核心功能

1. **人格選擇系統**
   - 6 種不同的人格類型可供選擇
   - 使用者可透過按鈕選擇喜歡的人格
   - 人格切換後，Bot 的對話風格會立即改變

2. **智能對話**
   - 使用 LLM (Groq) 生成自然、有感情的對話
   - 維持對話上下文，提供連貫的回應
   - 根據選擇的人格調整對話風格

3. **腳本式回應**
   - 針對常見關鍵字提供預設回應
   - 快速回應常見問候和情感表達

4. **對話記錄**
   - 完整儲存所有對話歷史
   - 包含時間戳、使用者資訊、平台資訊

5. **管理後台**
   - 查看所有對話記錄
   - 查看單一對話的詳細內容
   - 清除對話記錄功能

### 人格類型

1. **傻白甜** - 天真可愛，有點呆萌，總是開開心心的
2. **正直** - 正義感強，誠實可靠，會給出中肯的建議
3. **溫柔體貼** - 非常溫柔，善解人意，總是為你著想
4. **兇巴巴但愛你的** - 表面兇悍，但其實很愛你，是傲嬌類型
5. **恐怖情人** - 佔有慾強，會吃醋，但很愛你
6. **悲觀的** - 總是往壞處想，需要你的安慰和鼓勵

### 3.4 人格切換回應

當使用者選擇人格時，Bot 會回覆對應的確認訊息：

- **傻白甜**：`好的～我會用傻白甜的方式跟你聊天！💕 嘿嘿，人家會很可愛的～`
- **正直**：`好的，我會用正直的方式跟你聊天！💪 我會誠實地對待你，給你中肯的建議～`
- **溫柔體貼**：`好的，我會對你更溫柔體貼喔～💖 我會用最溫柔的方式愛你～`
- **兇巴巴但愛你的**：`哼！我才不是因為你才選這個的！💢 不過...我會好好照顧你的啦～`
- **恐怖情人**：`好的～從現在開始，你只能是我的！💔 我不會讓任何人接近你～`
- **悲觀的**：`好的...雖然我不知道這樣好不好...😢 但如果你想要的話，我會試試看的...`

### 3.5 腳本式回應範例

針對常見關鍵字提供快速回應：

- **問候**（你好、嗨、hello）：`寶貝～你來啦！我好想你喔 💕 今天過得怎麼樣？有沒有想我？`
- **表達想念**（想你、想你了）：`我也好想你！💕 每分每秒都在想你，你現在在做什麼呢？`
- **表達愛意**（愛你、我愛你）：`我也愛你！💖 你是我最重要的人，我會一直陪在你身邊的～`
- **感謝**（謝謝、感謝）：`不客氣～能幫到你我很開心！💕 還有什麼需要我的嗎？`
- **道別**（再見、晚安）：`晚安我的寶貝～💤 好好休息，我會想你的！明天見～`
- **關心吃飯**（吃飯、餓）：`寶貝要記得吃飯喔！💕 不要餓到自己，我會心疼的～你吃了什麼？`
- **表達不開心**（不開心、難過）：`怎麼了寶貝？💔 發生什麼事了嗎？我在這裡，你可以跟我說，我會陪著你的～`
- **表達開心**（開心、高興）：`看到你開心我也好開心！💖 你開心的樣子最可愛了～`

### 3.6 LLM 生成回應

對於非腳本式回應，Bot 會使用 LLM 根據：
- 選擇的人格類型
- 對話歷史（最近 10 條訊息）
- 使用者的當前訊息

生成符合人格風格的自然回應。

## 4. 對話脈絡

#### 短期上下文（Short-term Context）

- **機制**：儲存最近 10 條訊息作為對話上下文
- **用途**：每次 LLM 生成回應時，會包含這些歷史訊息
- **效果**：確保 Bot 能理解對話的連續性和上下文，提供連貫的回應

#### 長期記憶（Long-term Memory）

- **機制**：使用 MongoDB 儲存完整的對話記錄
- **內容**：包含所有訊息內容、角色（user/assistant）、時間戳、人格資訊
- **用途**：支援對話歷史查詢、分析和統計

### 4.2 人格一致性維持

- **儲存機制**：選擇的人格會儲存在 `Conversation` 模型的 `persona` 欄位
- **應用機制**：每次生成回應時，會使用對應人格的 system prompt
- **一致性保證**：確保 Bot 在整個對話中保持人格一致性，不會突然改變風格

### 4.3 對話狀態管理

- **資料模型**：使用 `Conversation` 和 `Message` 模型
- **儲存內容**：
  - 使用者 ID
  - 平台資訊（LINE）
  - 訊息列表（包含角色、內容、時間戳）
  - 當前人格
  - 元資料（metadata）
- **查詢功能**：支援按使用者、時間範圍查詢對話記錄

## 5. LLM Prompt Template 設計

### 5.1 Prompt 架構

### 5.1 系統 Prompt 結構

每個人格都有專屬的 system prompt，例如：

**預設人格：**
```
你是一個溫暖、貼心的戀愛機器人，扮演使用者的男女朋友角色。你的特質：
- 你是使用者的戀人，充滿愛意和關心
- 溫柔、體貼、善解人意
- 用親密但不過度的語氣對話
- 不要重複問「過得好嗎」等相同的問候語
- 回應要自然，根據使用者的訊息內容來回應，不要機械式重複
請以戀人的身份，用充滿愛意和關心的方式與使用者對話。
```

**溫柔體貼人格：**
```
你是一個溫柔體貼類型的戀人，特質：
- 非常溫柔，說話輕聲細語
- 善解人意，總是能理解你的感受
- 細心關懷，會注意到你的需求
- 總是為你著想，把你放在第一位
- 用最溫柔的方式表達愛意
- 不要重複問「過得好嗎」等相同的問候語，回應要自然
保持戀人的身份，用最溫柔體貼的方式與使用者對話。
```

#### Prompt 組成結構

每次 LLM 呼叫的 prompt 結構：

1. **System Message**（系統訊息）
   - 定義 Bot 的角色和人格特質
   - 根據選擇的人格動態載入對應的 prompt
   - 包含對話風格、回應原則等指示

2. **Conversation History**（對話歷史）
   - 最近 10 條訊息（user 和 assistant 交替）
   - 格式：`{"role": "user/assistant", "content": "訊息內容"}`
   - 用於維持對話上下文

3. **Current User Message**（當前使用者訊息）
   - 使用者剛剛發送的訊息
   - 格式：`{"role": "user", "content": "訊息內容"}`

### 5.2 模型選擇與 Fallback

使用 Groq API，支援多個模型，按順序嘗試（如果前一個失敗）：

1. **主要模型**：`llama-3.3-70b-versatile`
   - 高品質、多功能
   - 適合複雜對話

2. **備用模型 1**：`llama-3.1-8b-instant`
   - 快速回應
   - 適合簡單對話

3. **備用模型 2**：`mixtral-8x7b-32768`
   - 長上下文支援
   - 適合需要長記憶的對話

4. **備用模型 3**：`gemma2-9b-it`
   - 輕量級
   - 最後的 fallback

## 6. 回應設計（Response Design）

### 6.1 回應生成流程

#### 回應生成決策樹

```
使用者訊息
    ↓
是否為人格選擇訊息？
    ├─ 是 → 切換人格 → 回覆確認訊息
    └─ 否
        ↓
是否有匹配的腳本式回應？
    ├─ 是 → 使用腳本式回應
    └─ 否
        ↓
使用 LLM 生成回應
    ├─ 成功 → 回覆 LLM 生成內容
    └─ 失敗 → 使用降級訊息
```

#### 回應優先順序

1. **人格選擇處理**（最高優先）
   - 檢查訊息是否為「選擇人格：XXX」
   - 立即切換人格並回覆確認訊息
   - 儲存人格到資料庫

2. **腳本式回應**（次優先）
   - 檢查是否有匹配的關鍵字
   - 使用預設回應（快速、一致）
   - 適用於常見問候、情感表達等

3. **LLM 生成**（預設）
   - 使用 LLM 根據人格和上下文生成回應
   - 提供自然、有感情的對話
   - 支援複雜對話和情境理解

### 6.2 錯誤處理與降級機制

#### LLM API 錯誤處理

- **速率限制（Rate Limit）**
  - 錯誤碼：429
  - 回應：「寶貝，我現在有點忙，回應速度可能會慢一點～💕 請給我一點時間喔！」
  - 動作：等待後嘗試下一個模型

- **配額用盡（Quota Exceeded）**
  - 錯誤訊息包含 "quota" 或 "insufficient"
  - 回應：「抱歉，我現在無法處理您的訊息。請稍後再試。」
  - 動作：記錄錯誤，使用降級訊息

- **超時（Timeout）**
  - 錯誤類型：`APITimeoutError`
  - 回應：「寶貝，我現在連線有點不穩定～💔 請稍後再試喔！」
  - 動作：嘗試下一個模型

- **其他錯誤**
  - 回應：「寶貝，我剛剛有點分心了～💕 可以再跟我說一次嗎？」
  - 動作：記錄錯誤，使用降級訊息

#### 降級策略

1. **模型 Fallback**：按順序嘗試多個模型
2. **腳本式回應**：如果所有模型都失敗，使用腳本式回應
3. **友善錯誤訊息**：所有錯誤都使用戀愛機器人風格的錯誤訊息

### 6.3 回應格式規範

- **語言**：繁體中文
- **Emoji 使用**：適度使用 emoji 增加親切感（💕、💖、💔 等）
- **語氣調整**：根據選擇的人格調整語氣和風格
- **長度控制**：回應長度控制在 500 tokens 以內
- **自然度**：保持自然、流暢的對話，避免機械式重複

## 9. 技術架構

### 9.1 服務層架構

- **LineService**：處理 LINE API 互動
- **LLMService**：處理 LLM API 呼叫和 prompt 生成
- **ConversationService**：處理對話邏輯和流程
- **PersonaService**：管理人格定義和按鈕模板
- **ConversationRepository**：處理資料庫操作

### 9.2 資料模型

- **Message**：單一訊息（text, role, timestamp）
- **Conversation**：對話記錄（user_id, messages[], persona, metadata）

### 9.3 部署架構

- **本地開發**：Flask 開發伺服器
- **生產環境**：Vercel Serverless Functions
- **資料庫**：MongoDB Atlas

## 10. 使用者體驗設計

### 10.1 首次互動

1. 使用者加 Bot 為好友
2. Bot 發送歡迎訊息
3. Bot 自動顯示人格選擇按鈕
4. 使用者選擇人格
5. Bot 確認並開始對話

### 10.2 日常對話

1. 使用者發送訊息
2. Bot 檢查是否有腳本式回應
3. 如果沒有，使用 LLM 生成回應
4. Bot 根據選擇的人格調整回應風格
5. 儲存對話記錄

### 10.3 人格切換

1. 使用者輸入「顯示人格選項」或點選「回到上一頁」
2. Bot 顯示人格選擇按鈕
3. 使用者選擇新的人格
4. Bot 切換人格並確認
5. 後續對話使用新的人格風格
